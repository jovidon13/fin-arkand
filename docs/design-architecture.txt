03
Backend — архитектура (DRF)
модульный монолит
Архитектура — модульный монолит с сервисным слоем. Каждый домен холдинга это
отдельное Django-приложение; внутри приложения логика разложена по слоям. Финансы
требуют строгости: деньги, транзакции и права проверяются в одном месте, а не размазаны
по вьюхам.
Слои внутри модуля
Слой
Ответственность
Правило
models.py
структура данных, инварианты на уровне
записи
без бизнес-логики
selectors.py
чтение: выборки, фильтры, агрегации
только read
services.py
запись: вся бизнес-логика, транзакции
только здесь пишем в БД
serializers.py
валидация входа, форма ответа
без логики предметной области
views.py
тонкие ViewSets: принять → вызвать сервис →
отдать
никакой логики
permissions.py
RBAC + object-level доступ
роли и «своё/чужое»
Главное правило: бизнес-логика живёт в services.py и selectors.py. Views и serializers
остаются тонкими. Это даёт тестируемость, переиспользование (сервис зовётся и из API, и из
Celery, и из management-команды) и предсказуемость для команды.

│   ├── accounts/        # пользователи, роли, права
│   ├── finance/         # приходы/расходы, прибыль (ФНС-01…04)
│   ├── cash/            # кассы, лимиты оборота (КАС-01…04)
│   ├── settlements/     # взаиморасчёты, долги, бартер (БАР-01…04)

Разложение модуля (пример finance/)
Финансовые инварианты — обязательны
Только Decimal для денег — DecimalField(max_digits=14, decimal_places=2).
Никогда float.
Атомарность — любая денежная операция в transaction.atomic(); при гонках — 
select_for_update().
Идемпотентность — повтор запроса не создаёт вторую операцию (ключ
идемпотентности / проверка статуса).
Аудит — каждое изменение денег и статусов пишется в audit (кто, что, когда, до/после).
Soft-delete — финансовые записи не удаляются физически, а помечаются; для
отчётности и истории.
Изоляция касс — «касса видит только своё» реализуется фильтрацией queryset + object-
level permission, а не скрытием в UI.

class Transaction(TimeStampedModel):
    business    = models.ForeignKey("core.Business", on_delete=models.PROTECT)
    kind        = models.CharField(choices=TxKind.choices)      # income | expense
    category    = models.ForeignKey("finance.ExpenseCategory", null=True, 
on_delete=models.PROTECT)
    amount      = models.DecimalField(max_digits=14, decimal_places=2)  # деньги = Decimal
    method      = models.CharField(choices=PayMethod.choices)   # cash | transfer
    status      = models.CharField(default=TxStatus.PENDING)
    confirmed_by= models.ForeignKey(settings.AUTH_USER_MODEL, null=True, 
on_delete=models.SET_NULL)
# settlements/services.py — авто-фиксация долга при одобренной передаче (ХОЛ-30)
@transaction.atomic
def approve_transfer(*, transfer: Transfer, actor: User) -> Debt:
    transfer.mark_approved(actor)                    # смена статуса
    debt = Debt.objects.create(
        debtor=transfer.to_business, creditor=transfer.from_business,
        amount=transfer.amount, source=transfer,     # прозрачный реестр
    )
    AuditLog.record(actor, "transfer.approved", transfer)
return debt

04
Admin — архитектура (React / FSD)
Feature-Sliced Design
Admin-панель строится по Feature-Sliced Design: код разложен по слоям от общего к
частному, импорты идут строго сверху вниз. Это масштабируется под холдинг — новый
модуль добавляется как набор слайсов, не задевая остальные.
Слои FSD
Слой
Что внутри
Пример для ARKAND
app
провайдеры, роутер,
тема, i18n, глобальные
стили
токены, ProtectedRoute, layout
pages
страницы-композиции из
виджетов
Финансы, Кассы, Долги, Зарплата, Отчёты

(сценарии)

бизнес-сущности и их
представление
transaction, cash-register, debt, employee, business

кнопки, поля, токены, axios, форматтеры денег
Правило импортов: слой видит только слои ниже себя. pages → widgets → features →
entities → shared. Слайсы на одном слое между собой не импортируются напрямую. Это и есть
защита от «спагетти».

05
Параллельная разработка
как делить работу
Модульность backend и FSD frontend позволяют команде работать параллельно с
минимумом конфликтов. Точка стыковки — контракт API: как только он согласован, backend
и frontend движутся независимо.
Backend — по доменам
Разработчик A — core + accounts +
права (фундамент для всех).
Разработчик B — finance + cash
(приходы/расходы, кассы, лимиты).
Разработчик C — settlements + payroll +
reports (долги, зарплата, отчёты).
Frontend — по фичам
Общий shared/ui + токены делает один
человек первым.
Дальше страницы берутся параллельно:
Финансы, Кассы, Долги, Зарплата,
Отчёты.
Сущности (entities/*) переиспользуются
между страницами.
Что согласовать до старта кода: Часть 0 (список бизнесов, пользователи, роли), правила
ХОЛ-30…33 (логика долгов и взаиморасчётов) и формат данных для надстройки владельцев
(Часть 7). Дизайн-система и стек в этом документе уже зафиксированы.
